From the scatter plot, it can be observed that there is no obvious linear relationship between the length of positive reviews and the reviewer scores.

Analysis: There may be several reasons why users give positive reviews with fewer words:
1) Guests have high satisfaction, which does not require much description. It is possible that guests who have had a positive experience with the hotel may leave shorter reviews because they may not feel the need to provide as much detail as they would in a negative review. It is common for people to provide more specific feedback when they have had a negative experience. Therefore, a shorter positive review does not necessarily mean that it is less valuable or unreliable.
2) Time constraints that prevent them from writing longer reviews. However, a shorter positive review does not necessarily mean that it is false or has no value, as long as the review is genuine and not fake, it indicates that the hotel is relatively reliable. Further analysis of the positive reviews using text analysis techniques can provide insights into what aspects of the hotel guests are satisfied with.

Some quick comments regarding option 2

Storing all information in one wide table has the following disadvantages:

  1. Occupies large storage space due to a large number of columns in the table.
  2. May contain a lot of data redundancy, such as high duplication rates of hotel addresses and longitude/latitude information.
  3. Poor maintainability. When the data structure and schema change, all related columns need to be updated.

We can use the Snowflake modeling concept to split the big table into three tables: hotel dimension table (hotel), reviewer dimension table (reviewer), and fact table (review). By doing this, we can eliminate data redundancy and reduce the data storage to the maximum extent. Additionally, joining the smaller dimension tables can improve query performance.

# Hotel dimension table (hotel)
CREATE TABLE hotel (
    hotel_id INTEGER PRIMARY KEY COMMENT 'Hotel ID',
    hotel_name varchar(255) COMMENT 'Hotel name',
    hotel_address varchar(255) COMMENT 'Hotel Address',
    lat FLOAT COMMENT 'Latitude',
    lng FLOAT COMMENT 'Longitude'
);

# Reviewer dimension table (reviewer)
CREATE TABLE reviewer (
    reviewer_id INTEGER PRIMARY KEY COMMENT 'Reviewer ID',
    reviewer_name varchar(255) COMMENT 'Reviewer name',
    nationality varchar(50) COMMENT 'Reviewer nationality'
);

# Fact table (review)
CREATE TABLE review (
    review_id INTEGER PRIMARY KEY COMMENT 'Review ID',
    hotel_id INTEGER COMMENT 'Hotel ID',
    user_id INTEGER COMMENT 'Reviewer ID',
    review_date DATETIME COMMENT 'Review date',
    days_since_review varchar(50) COMMENT 'Days since review',
    reviewer_score FLOAT COMMENT 'Reviewer Score',
    negative_review text COMMENT 'Negative review',
    positive_review text COMMENT 'Positive review'
);

# Finally, the positive reviews, negative reviews, and tags were merged based on the review date and reviewer country through association.
# The average score was calculated by aggregating the review scores, and the original table was obtained.
select hotel_address, review_date, hotel_name, days_since_review, reviewer_nationality, lat, lng,
group_concat(negative_review) negative_review, 
group_concat(positive_review) positive_review, 
avg(reviewer_score) reviewer_score,
group_concat(tags) tags, 
count(c.user_id) total_number_of_reviews_reviewer_has_given
from review a
join hotel b on a.hotel_id=b.hotel_id
join reviewer c on a.user_id=c.user_id
group by hotel_address, review_date, hotel_name, days_since_review, reviewer_nationality, lat, lng
